# DTLN-aec model spec (256-variant)

DTLN-aec is a two-stage LSTM acoustic echo canceller from Westhausen +
Meyer (Interspeech 2021, *"Acoustic echo cancellation with the dual-signal
transformation LSTM network"*). Upstream training code and TFLite weights
live at <https://github.com/breizhn/DTLN-aec> under the MIT license. This
document is the 256-variant spec — doubled LSTM hidden size vs the
originally-researched 128-variant — and the ground truth for the Rust
port in `crates/aec`.

## Source

- Original weights: <https://github.com/breizhn/DTLN-aec> (MIT,
  Westhausen et al.). Upstream ships TFLite checkpoints.
- ONNX export: the 256-variant TFLite weights converted once and
  committed in this directory (`model_256_1.onnx`, `model_256_2.onnx`).
- This document is the 256-variant adaptation of the originally-researched
  128-variant spec.

## Model files

- Number of `.onnx` files the runtime loads: **2** (two-stage pipeline)
- File names for the **256-variant** (used by this Rust port):
  - `model_256_1.onnx` — Stage 1, ~5.6 MB
  - `model_256_2.onnx` — Stage 2, ~10.0 MB
  - Total: ~15.6 MB
- Other size variants exist upstream (not used here):
  - `model_128_1.onnx` (~2.0 MB), `model_128_2.onnx` (~5.3 MB) — upstream default
  - `model_512_1.onnx` (~17.4 MB), `model_512_2.onnx` (~24.2 MB)
- Checksums of the committed copies:
  - `model_256_1.onnx`: `e7df5a25fce3565bc7d3b9c0b3a4e53f7adb96a4cc772dc189df895f6f33adb7`
  - `model_256_2.onnx`: `19c07ebe462c424523609afb635e5ff42f1127aa434879ddb744d2db68a22dee`
- **Reproducing the files**:
  - **Convert from upstream yourself** —
    `bash models/dtln-aec/convert-from-upstream.sh`. Downloads the MIT
    TFLite weights from <https://github.com/breizhn/DTLN-aec> and converts
    them locally with a pinned tf2onnx (needs Python 3.9–3.11, or
    podman/docker for the containerized fallback). Output is functionally
    identical to the release copies — same graph and node names, passes the
    `aec` crate test suite (`cargo test -p aec`) — but not byte-identical,
    so the SHA-256 sums above don't apply to converted files.

## Sample rate, frame size, blocking

| Constant     | Value  | Notes                                            |
|--------------|--------|--------------------------------------------------|
| Sample rate  | 16 000 Hz | Confirmed in `audacity.rs` (`load_mono_16khz()`) |
| `BLOCK_SIZE` | 512 samples | Full FFT window; 32 ms at 16 kHz              |
| `BLOCK_SHIFT`| 128 samples | Hop size; 75% overlap; 8 ms at 16 kHz        |
| FFT bins     | 257    | `BLOCK_SIZE/2 + 1 = 257`                         |
| `STATE_SIZE` | **256**| LSTM hidden dimension (256-variant)              |
| Window fn    | None   | Raw buffer fed directly to FFT                   |

## Stage 1 (ONNX inference)

Stage 1 takes frequency-domain magnitude spectra of both the mic (near-end)
and loudspeaker (far-end) signals and outputs a real-valued suppression mask
over frequency bins.

### Inputs

| Slot | ONNX name | Shape | Dtype | Notes |
|------|-----------|-------|-------|-------|
| 0 | `input_3` | `(1, 1, 257)` | float32 | Magnitude spectrum of mic signal; 257 = `BLOCK_SIZE/2+1`; zeros at first frame |
| 1 | `input_5` | `(1, 2, 256, 2)` | float32 | LSTM hidden+cell state; zeros at first frame; shape `(batch=1, layers=2, state_size=256, h_c=2)` |
| 2 | `input_4` | `(1, 1, 257)` | float32 | Magnitude spectrum of loudspeaker (far-end); zeros at first frame |

> **Note on slot names**: `ort 2.x` `inputs!` macro passes tensors
> positionally by slot index. The names above (`input_3`, `input_5`,
> `input_4`) are the autogenerated ONNX graph node names observed when
> loading `model_256_1.onnx` via `ort`. Use slot index for
> the Rust code; use `session.get_inputs()[i].name` in any Python tooling.

### Outputs

| Slot | ONNX name | Shape | Dtype | Notes |
|------|-----------|-------|-------|-------|
| 0 | `Identity` | `(1, 1, 257)` | float32 | Real-valued suppression mask; applied element-wise to mic FFT complex bins |
| 1 | `Identity_1` | `(1, 2, 256, 2)` | float32 | Updated LSTM states; fed back as slot 1 (`input_5`) on the next frame |

### State carried between frames

- `states_1` (slot 1, name `input_5`): shape `(1, 2, 256, 2)`, dtype float32 —
  LSTM hidden and cell states for Stage 1. Initialized to zeros. Output
  `Identity_1` from each call becomes input slot 1 for the next call.

## Stage 2 (ONNX inference)

Stage 2 refines the time-domain signal produced by Stage 1's mask
application. It operates entirely in the time domain.

### Inputs

| Slot | ONNX name | Shape | Dtype | Notes |
|------|-----------|-------|-------|-------|
| 0 | `input_6` | `(1, 1, 512)` | float32 | Time-domain block after mask+IFFT; normalized by `1.0/BLOCK_SIZE`; zeros at first frame |
| 1 | `input_8` | `(1, 2, 256, 2)` | float32 | LSTM hidden+cell state for Stage 2; zeros at first frame |
| 2 | `input_7` | `(1, 1, 512)` | float32 | Raw time-domain loudspeaker (far-end) block from `in_buffer_lpb`; zeros at first frame |

> **Note**: Stage 2 uses distinct autogenerated node names (`input_6`,
> `input_8`, `input_7`) from Stage 1 (`input_3`, `input_5`, `input_4`), as
> observed from loading `model_256_2.onnx` via `ort`. Use slot
> index for the Rust code; use `session.get_inputs()[i].name` in Python.

### Outputs

| Slot | ONNX name | Shape | Dtype | Notes |
|------|-----------|-------|-------|-------|
| 0 | `Identity` | `(1, 1, 512)` | float32 | Cleaned time-domain audio block (512 samples) |
| 1 | `Identity_1` | `(1, 2, 256, 2)` | float32 | Updated LSTM states; fed back as slot 1 (`input_8`) on the next frame |

### State carried between frames

- `states_2` (slot 1, name `input_8`): shape `(1, 2, 256, 2)`, dtype float32 —
  LSTM hidden and cell states for Stage 2. Initialized to zeros. Output
  `Identity_1` from each call becomes input slot 1 for the next call.

## Pre-processing

Steps applied between raw f32 audio in and tensor input to Stage 1
(per the DTLN-aec inference loop):

1. **Audio format**: Inputs arrive as `&[f32]` slices in the range `[-1, 1]`.
   No int16→float32 conversion is done inside the AEC crate. The caller
   resamples to 16 kHz and downmixes to mono before passing in. In Daisy's
   pipeline, PipeWire captures float32 directly; int16 conversion (multiply
   by 32767.0, clamp, cast) only happens at the downstream storage boundary.

2. **Framing via circular buffer**: Two circular buffers of length
   `BLOCK_SIZE=512` with shift `BLOCK_SHIFT=128` are maintained — one for
   mic (near-end) and one for loudspeaker (far-end). Each call to
   `process_streaming` consumes `BLOCK_SHIFT=128` new samples per channel:
   ```
   in_buffer.push_chunk(&audio[start..end])       // 128 mic samples
   in_buffer_lpb.push_chunk(&lpb[start..end])     // 128 far-end samples
   ```
   `push_chunk` shifts the 512-sample buffer left by 128 and appends the
   new 128 samples at the end (oldest discarded).

3. **No window function**: The 512-sample buffer is fed directly to FFT with
   no windowing (no Hann, no sqrt-Hann). Confirmed against the original DTLN-aec inference code.

4. **FFT**: The `rustfft` crate (or `realfft` for the real-valued variant)
   computes forward real FFT of the 512-sample buffer. Output is 257 complex
   bins (`BLOCK_SIZE/2 + 1`).

5. **Magnitude extraction**: For each complex bin `c` in the 257-element FFT
   result:
   ```
   in_mag[0, 0, i] = c.norm()    // sqrt(re² + im²)
   ```
   Shapes: `in_mag = (1,1,257)`, `lpb_mag = (1,1,257)`.

6. **FFT phase saved**: The complex FFT result (`in_block_fft`) is retained
   for mask application after Stage 1 returns.

## Post-processing

Steps applied between Stage 2 tensor output and final PCM frame
(per the DTLN-aec inference loop):

1. **Mask application** (between Stage 1 and Stage 2): Stage 1 outputs a real
   mask of shape `(1,1,257)`. Applied element-wise to the saved complex FFT
   bins:
   ```
   for (c, m) in in_block_fft.iter_mut().zip(out_mask) {
       *c *= m   // complex *= real scalar
   }
   ```

2. **IFFT**: Inverse real FFT on the 257 masked complex bins → 512 real
   samples.

3. **IFFT normalization**: Output is scaled by `1.0 / BLOCK_SIZE`:
   ```
   estimated_block = ifft_output * (1.0 / 512.0)
   ```
   This normalized `(1,1,512)` tensor is the input to Stage 2.

4. **Stage 2 output**: `(1,1,512)` time-domain block from Stage 2 `Identity`
   output is copied to the output context.

5. **Overlap-add via `out_buffer.shift_and_accumulate`**:
   - The output buffer (length 512) is shifted left by 128.
   - The newly vacated trailing 512-sample region is zeroed.
   - The Stage 2 output (512 samples) is **added** (not replaced) into the
     buffer starting at index 0.
   - Every call, the first `BLOCK_SHIFT=128` samples of the output buffer
     are ready as the output frame.

6. **Peak normalization** (batch mode only): After all frames are accumulated,
   if `max(|output|) > 1.0`, scale entire output by `0.99 / max(|output|)`.
   Not applied in streaming mode.

7. **Output format**: Remains float32. For downstream storage or transmission
   as int16, multiply by 32767.0, clamp to `[-32768, 32767]`, cast to `i16`.

## Far-end alignment

- The upstream code **assumes the far-end (loudspeaker) signal is
  sample-aligned with the near-end (mic) signal**. There is no delay
  estimation, ring buffer for alignment, or cross-correlation step.
  `push_chunk` is called for both buffers in the same loop iteration,
  consuming the same index range.
- Real-world AEC performance degrades significantly if loopback latency is
  not corrected externally. In Daisy's pipeline, loopback audio from
  PipeWire may lag the mic by a platform-dependent amount (typically 10–100
  ms). A fixed-delay circular buffer should be added in the Rust wrapper if
  initial tests show residual echo.
- All buffers (`in_buffer`, `in_buffer_lpb`, `out_buffer`) are 512-sample
  rings with 128-sample hop.

## Performance notes

- **Runtime**: `ort` version `2.0.0-rc.10` (pinned in `Cargo.toml`).
- **Execution providers**: CPU is always available. Optional features for
  `cuda`, `coreml`, and `directml` backends are compile-time opt-ins.
- **No benchmarks published**: No performance numbers appear in the upstream
  README or test suite. DTLN-aec with 256-unit state is heavier than the
  128-variant; real-time on a single CPU core at 16 kHz is expected but
  should be validated on the target machine.
- **Realtime factor**: The 128-variant is the smallest and fastest. The
  256-variant approximately doubles the LSTM weight count (roughly 2× FLOPs
  per frame). Benchmark with `cargo bench -p aec` once implemented.

## Algorithm pseudocode (Rust port)

This is not Python — it describes the algorithm in pseudocode form. The
actual implementation lives in `crates/aec/src/`.

```rust
// Constants
const BLOCK_SIZE:  usize = 512;
const BLOCK_SHIFT: usize = 128;
const STATE_SIZE:  usize = 256;  // 256-variant
const FFT_BINS:    usize = BLOCK_SIZE / 2 + 1;  // 257

// State (zero-initialized, carried across frames)
let mut near_buf:  [f32; BLOCK_SIZE] = [0.0; BLOCK_SIZE];
let mut far_buf:   [f32; BLOCK_SIZE] = [0.0; BLOCK_SIZE];
let mut out_buf:   [f32; BLOCK_SIZE] = [0.0; BLOCK_SIZE];
let mut states1: Array4<f32> = Array4::zeros((1, 2, STATE_SIZE, 2));
let mut states2: Array4<f32> = Array4::zeros((1, 2, STATE_SIZE, 2));

// Per-frame (called once per BLOCK_SHIFT=128 new samples)
fn process_frame(near_chunk: &[f32; 128], far_chunk: &[f32; 128]) -> [f32; 128] {
    // 1. Push into circular buffers (shift-left + append)
    near_buf.rotate_left(BLOCK_SHIFT);
    near_buf[BLOCK_SIZE - BLOCK_SHIFT..].copy_from_slice(near_chunk);
    far_buf.rotate_left(BLOCK_SHIFT);
    far_buf[BLOCK_SIZE - BLOCK_SHIFT..].copy_from_slice(far_chunk);

    // 2. Forward FFT (no window)
    let near_fft: [Complex<f32>; FFT_BINS] = rfft(&near_buf);
    let far_fft:  [Complex<f32>; FFT_BINS] = rfft(&far_buf);

    // 3. Magnitude spectra
    let in_mag:  Array3<f32> = near_fft.iter().map(|c| c.norm()).collect(); // (1,1,257)
    let lpb_mag: Array3<f32> = far_fft.iter().map(|c| c.norm()).collect();  // (1,1,257)

    // 4. Stage 1: echo-suppression mask
    let (mask, new_states1) = stage1.run(in_mag, states1, lpb_mag);
    states1 = new_states1;  // carry forward

    // 5. Apply mask in frequency domain
    let masked_fft: [Complex<f32>; FFT_BINS] = near_fft
        .iter().zip(mask.iter())
        .map(|(c, m)| c * m)
        .collect();

    // 6. IFFT + normalize by 1/BLOCK_SIZE
    let time_domain: [f32; BLOCK_SIZE] = irfft(&masked_fft);
    let estimated: Array3<f32> = time_domain.map(|x| x / BLOCK_SIZE as f32); // (1,1,512)

    // 7. Far-end time-domain block for Stage 2
    let in_lpb: Array3<f32> = Array3::from(far_buf.to_vec()); // (1,1,512)

    // 8. Stage 2: time-domain refinement
    let (out_block, new_states2) = stage2.run(estimated, states2, in_lpb);
    states2 = new_states2;  // carry forward

    // 9. Overlap-add into output buffer
    out_buf.rotate_left(BLOCK_SHIFT);
    out_buf[BLOCK_SIZE - BLOCK_SHIFT..].fill(0.0);
    for (o, s) in out_buf.iter_mut().zip(out_block.iter()) {
        *o += s;
    }

    // 10. Emit first BLOCK_SHIFT samples
    let mut output = [0.0f32; BLOCK_SHIFT];
    output.copy_from_slice(&out_buf[..BLOCK_SHIFT]);
    output
}
```

> **IFFT normalization note**: `realfft`'s `process_with_scratch` does NOT
> normalize (unlike NumPy's `irfft` which divides by `n`). The explicit
> `* (1.0 / BLOCK_SIZE)` above matches the original DTLN-aec inference. Do not
> double-normalize.

## Open questions (carried forward from 128-variant research)

1. **ONNX input node names**: Autogenerated names confirmed by loading the
   `model_256_*.onnx` files via `ort`:
   - Stage 1 inputs: `input_3` (slot 0), `input_5` (slot 1), `input_4` (slot 2)
   - Stage 2 inputs: `input_6` (slot 0), `input_8` (slot 1), `input_7` (slot 2)
   - Both stages output `Identity` (slot 0) and `Identity_1` (slot 1).
   Update this doc if upstream retrains the models (names may change).

2. **Far-end alignment for Daisy**: the reference implementation assumes pre-aligned near/far
   signals. Daisy's PipeWire loopback may lag the mic by 10–100 ms. Add a
   fixed-delay buffer in `crates/aec/src/` if initial integration tests show
   residual echo.

3. **Output shape of Stage 2 `Identity`**: Confirmed `(1, 1, 512)` by an
   `ort` probe.

4. **State shape**: `(1, 2, 256, 2)` confirmed by `ort` probe.
   Last dimension interleaves LSTM `h` and `c` states.

5. **Git LFS**: As of the commit referenced above, the `.onnx` files are
   regular git blobs (not LFS pointers).
